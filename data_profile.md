## 3. Signal audit

This is synthetic data. Everything below is measured, and it changes what gets built.

### 3.1 Conversion target

`sessions.is_converted`: **base rate 7.46%** (1,440 / 19,315). Imbalanced — accuracy is a useless metric here, so the model is scored on ROC-AUC / PR / recall / F1 and the confusion matrix.

### 3.2 Leakage — found and excluded

**`purchases` is a perfect leak.** Verified: `set(purchases.session_id) == set(sessions[is_converted].session_id)`, exactly. A session is converted **iff** it has a purchase row. Any feature derived from `purchases` (revenue, order count, `interaction_id` join) predicts the target with 100% accuracy and is unavailable at prediction time.

→ **The entire `purchases` table is excluded from conversion features.** It is used only for revenue KPIs, segmentation (AOV), and CF purchase weighting.

Relatedly, the spec's proposed `purchase = 5` interaction weight **cannot be applied as written** — there is no `purchase` interaction type. Purchase signal is joined in from the `purchases` table instead (see README).

### 3.3 `add_to_cart` is a one-way perfect separator — kept, but disclosed

This is the single most important finding:

| net cart-adds in session | P(converted) | n sessions |
|---|---|---|
| 0 | **0.0000** | 10,944 |
| 1 | 0.0477 | 5,892 |
| 2 | 0.4073 | 1,736 |
| 3 | 0.5762 | 538 |
| 4+ | 0.6927 | 205 |

**A session with zero cart-adds converts zero times out of 10,944.** That is a deterministic rule in the generator, not a tendency.

Judgment call, stated plainly: this is **not** leakage and is **kept**. `add_to_cart` causally precedes purchase and is genuinely observable mid-session (unlike `purchases`), so a real-time model would legitimately have it. But it means **the conversion model is substantially just learning "did they add to cart, and how many times"**. The headline ROC-AUC is therefore real but not impressive engineering — it reflects a generator rule. This is disclosed in the README's limitations section rather than being sold as model quality.

### 3.4 Features with no signal — measured, kept only to prove the point

| feature | conversion rate spread | verdict |
|---|---|---|
| `device_type` | desktop 7.39% / mobile 7.53% / tablet 7.26% | **noise** |
| `referrer_source` | 7.24% – 8.05% across 7 sources (`display_ad` n=385) | **noise**, within sampling error |
| `dwell_time_ms` | converted **18,060** vs non-converted **19,617** (mean) | **noise**, and slightly inverted |

The generator did not condition conversion on device, referrer, or dwell time. These are included in the feature set anyway (they are real, non-leaky session attributes), and the trained feature-importance chart is expected to show them near zero — which is itself an honest result worth displaying on `/model`.

Genuine secondary signal exists in engagement volume: converted sessions average **7.60** interactions vs **4.98** for non-converted, and `click` lifts conversion 7.46% → 10.04%.

### 3.5 Interactions are not uniform noise — CF is viable

| grouping | mean | median | max | std |
|---|---|---|---|---|
| interactions per session | 5.18 | 4 | 28 | 3.88 |
| interactions per user | 14.40 | 11 | 88 | 12.06 |
| interactions per product | 103.41 | 20 | **6,777** | 390.82 |

Per-product distribution is **heavily skewed** (median 20, max 6,777, std 4× the mean) — a real long-tail popularity structure, not uniform random assignment. Collaborative filtering has something to latch onto.

Caveat: with 967 items × 6,944 users over 100K interactions, the matrix is ~1.5% dense. Co-occurrence support for tail items is thin, so item-item similarity for rare products rests on few observations.

### 3.6 Temporal consistency

Sessions/interactions span 2023-01-04 → 2026-05-31 and align. But **reviews start 2022-01-13, a full year before the first session exists**, and 200 reviews have no `purchase_id`. Reviews were generated on an independent timeline and cannot be causally chained back to sessions. Reviews are therefore used only for product-level rating aggregates and segmentation counts — never for session-level conversion features.

## 4. Decisions carried into downstream code

1. Interaction weights remapped to the **real** vocabulary (no `purchase` type; negative signals demoted):
   `view=1, click=2, add_to_wishlist=3, add_to_cart=4`, purchase joined from `purchases`**=5**, `remove_from_cart`/`remove_from_wishlist` **negative**.
2. **All `purchases`-derived features excluded** from the conversion model (perfect leak).
3. `rating_avg` NULL preserved as "unrated", never imputed to 0.
4. Popularity fallback is **mandatory**, not a nicety — it serves 31% of users.
5. Honest metrics only. `/model` shows the real confusion matrix and the near-zero importances for device/referrer/dwell.
